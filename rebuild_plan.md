# 树莓派人脸管线重构方案

## 目标

在 Raspberry Pi 4 和 USB 摄像头的约束下，优先保证以下目标：

- 先保证画面预览流畅、交互不卡顿
- 再保证人脸识别稳定可靠
- 情绪识别作为低优先级旁路任务
- 云端能力只用于处理困难样本，不阻塞本地实时画面

## 目标架构

```text
USB 摄像头
  -> 低延迟采集线程
  -> 最新帧队列（旧帧直接丢弃）
  -> 轻量级人脸检测器（带关键点）
  -> 短时跟踪器
  -> 人脸对齐模块（5点相似变换）
  -> 识别调度器
       -> 对 unknown / 新轨迹 / 低置信度轨迹 做本地识别
       -> 对已稳定识别的轨迹，在冷却时间内复用身份结果
  -> 情绪调度器
       -> 只对稳定正脸、低频执行情绪识别
  -> 渲染 / 日志 / 存储
  -> 可选的云端异步复核模块（用于 unknown 或模糊样本）
```

## 哪些模块值得保留

- `CameraManager`
  - 保留当前生产者线程和“只取最新帧”的结构。
  - 后续扩展为更适合树莓派 USB 摄像头的 GStreamer 低延迟接入方式。
- `Database`、`FileStore`、`EmbeddingStore`
  - 保留目前的持久化设计和录入数据组织方式。
- `TrackManager`
  - 保留当前身份平滑、情绪平滑的思路。
- `Renderer`
  - 保留为本地调试和操作界面。

## 哪些模块需要替换或重点升级

- `FaceDetector`
  - 从“只输出框”的检测器升级为“输出框 + 人脸关键点”的检测器。
  - 优先考虑轻量级、适合边缘端的 5 点或 6 点关键点模型。
- `FaceRecognizer`
  - 保留当前 NCNN 推理路线，但输入不再直接使用原始裁剪框，而是改为“对齐后的人脸”。
- `InferencePipeline`
  - 从当前“按固定帧间隔触发”的逻辑，重构为“围绕轨迹状态调度”。
  - 识别应由事件驱动，而不是简单按取模：
    - 新出现的轨迹
    - 最近还是 unknown 的轨迹
    - 图像质量刚刚变好的轨迹
    - 云端结果刚返回的轨迹
- `EmotionRecognizer`
  - 可以保留，但要降优先级。
  - 只对主人脸或少数稳定轨迹执行，不再和主识别链路抢实时预算。

## 建议新增的模块

- `engine/FaceAligner`
  - 输入：原始图像 + 人脸关键点
  - 输出：对齐后的 `112x112` 人脸图，供识别模型使用
- `tracking/TrackPolicy`
  - 决定某条轨迹当前是否需要：
    - 刷新检测
    - 本地识别
    - 刷新情绪
    - 提交云端复核
- `engine/RecognitionCache`
  - 记录每条轨迹的身份缓存、冷却时间和最近识别状态
- `cloud/CloudVerifier`（可选）
  - 负责异步上传、等待返回和结果回写
  - 只处理 unknown、低 margin 或需要人工复核的样本

## 在当前仓库中的映射关系

- 建议保留：
  - `src/camera`
  - `src/storage`
  - `src/ui`
  - `src/tracking` 的大部分结构
- 建议重点重构：
  - `src/engine/FaceDetector.cpp`
  - `src/engine/FaceRecognizer.cpp`
  - `src/engine/InferencePipeline.cpp`
- 建议新增：
  - `src/engine/FaceAligner.cpp`
  - `include/engine/FaceAligner.hpp`
  - `src/tracking/TrackPolicy.cpp`
  - `include/tracking/TrackPolicy.hpp`
  - 如果后续接云端，再新增相关 `cloud` 模块

## 运行策略

### 检测

- 不要每帧都做人脸检测。
- 建议每 `4-6` 帧做一次完整检测。
- 如果跟踪置信度下降，提前触发一次检测刷新。

### 跟踪

- 在非检测帧之间，用轻量级跟踪器更新人脸框位置。
- 这一层是“跳帧检测”真正转化为 FPS 收益的关键。

### 识别

- 不要对所有候选框机械地按固定间隔识别。
- 只在以下情况下触发识别：
  - 新轨迹出现
  - 某轨迹连续 unknown，且冷却时间已过
  - 某张脸刚刚变大、变清晰、质量提升
  - 上一次结果是模糊匹配或歧义结果

### 情绪识别

- 限制为主人脸或极少量稳定轨迹。
- 执行频率控制在 `2-5 Hz`，不要跟随相机帧率全速运行。

### 云端协同

- 云端结果绝不能阻塞本地预览。
- 只上传这些情况：
  - unknown
  - top1 和 top2 很接近、margin 很小
  - 用户主动要求高精度复核

## 推荐实施顺序

1. 给检测结果补充关键点，并新增 `FaceAligner`
2. 把识别输入从原始裁剪框改成对齐后人脸
3. 加入“按轨迹状态调度”的识别策略和冷却时间机制
4. 在检测帧之间加入真正的轻量跟踪器
5. 把情绪识别限制为稳定主轨迹的低频任务
6. 最后再接入可选的云端异步复核模块

## 当前最容易先拿到收益的点

- 把树莓派端构建切换为 `Release`
- 运行时关闭逐帧调试日志
- 使用低延迟 USB 摄像头输入管线
- 本地继续用轻量识别模型，但通过“人脸对齐”提升输入质量
- 先用“每人模板向量”粗筛，再做样本级精排

## 为什么这套方案比“只换更大模型”更好

- 人脸对齐能在不显著增加推理成本的情况下明显提升识别准确率
- 按轨迹状态调度能减少大量无意义识别计算
- 低延迟采集能避免“推理一慢，画面越积越卡”的连锁问题
- 云端只处理真正困难的样本，最能发挥它的价值
